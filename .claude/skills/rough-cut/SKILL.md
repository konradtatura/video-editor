# Rough Cut

Raw recording -> transcript -> editorial cutlist (read by Claude, not a rules engine) -> cached-segment render -> mandatory re-verification against the actual rendered file.

## Why this shape

Every step here exists because a simpler version of it failed during development, on real footage, in a reproducible way. The two failure classes that matter most:

1. **Word timestamps lie near pauses.** WhisperX/Whisper's forced alignment assigns unclaimed silent frames to the *last* word before a pause, so a word's `end` timestamp can be 0.3-1.5s later than the word actually stops being spoken. Cutting at the literal word-end timestamp bakes dead air into the clip. Fix: never trust ASR end-timestamps for a cut boundary -- measure the real audio energy instead (`audio_boundaries.py`).

2. **Whisper suppresses genuine repeats, sometimes completely -- and this cuts both ways, which is the more dangerous direction.** When a speaker stutters or restarts a line, Whisper's decoder (trained on clean captions, and carrying an anti-hallucination heuristic that discards "too repetitive" output) can silently delete an entire repetition from the transcript text -- not mistime it, delete it, with zero trace. This got worse the smaller the model: `medium` missed repeats that `large-v3` caught on the identical audio. Chunking into short independently-decoded pieces helps but does **not** fully fix it, especially for rapid single-word stutters with no acoustic pause between repeats (verified: a 4x "jeśli chcesz" stutter with zero silence between attempts survived medium-model chunked transcription intact, and was only caught by re-running `large-v3` on the same audio). **Multiple independent transcription passes agreeing is not proof a repeat doesn't exist** -- confirmed on real footage: `chunk_transcribe.py`, `transcribe2.py`, and a fresh wide-context re-transcription all agreed on a single clean sentence where the audio actually had 3 false-start stutters folded in, because all three suffered the identical suppression on the identical audio. The other direction is just as real: **isolating a short (~1s) audio clip for re-transcription without surrounding context makes Whisper hallucinate plausible text instead of transcribing accurately** -- confirmed directly: two *different* ~1s bursts both "transcribed" to the identical phrase. Neither failure mode is rare or contrived; both happened in the same session on the same video. This is why acoustic (non-ASR) verification exists now -- see `bursts.py` / `find_repeats.py` below -- text-only cross-checking cannot catch either direction.

**The practical consequence: no transcript, at any settings, is ground truth, and neither is a short isolated re-transcription used to "double check" a suspicious span.** The reliable checks are (a) re-transcribing the actual rendered output file with full context and looking for what shouldn't be there, (b) measuring the actual audio -- burst durations and DTW acoustic similarity -- which cannot hallucinate and cannot suppress, and (c) a second, architecturally different ASR pass (`ctc_check.py`, see below) that doesn't share Whisper's suppression bias at all. This is why `verify.py` exists and why it is not optional. **If the user directly reports something is wrong at a specific moment, that outranks every automated signal here** -- use it to locate the fix (`locate.py`), don't re-litigate whether the problem is real by throwing more transcription at it.

3. **Whisper's suppression bias survives every mitigation that stays within Whisper -- the fix had to be a different decoder architecture.** Two acoustic-only approaches were tried and fell through before landing on this: within-a-span self-similarity (a real repeat's average similarity measured *lower* than a genuinely non-repeated control on real footage -- speakers change pace on a retake, so a true repeat isn't acoustically identical enough for a clean threshold) and front-half-vs-back-half subsequence matching on a single suspect span (no fused repeat crossed the match threshold; the burst boundary `find_repeats.py`'s deep-scan relies on turned out to be doing real, load-bearing work that a boundary-free version couldn't replicate). The actual fix: `ctc_check.py` runs a wav2vec2 CTC model (`ctc_backend.py`) as a full second pass over the raw audio. CTC has no equivalent bias -- it emits one token per audio frame independently, so two genuinely separate utterances of the same word remain two separate spans through the standard CTC collapse rule, gapless or not. Confirmed on real footage: it recovered both a 2x and a 3x fused repeat that Whisper (via Groq) had collapsed to one instance each -- including one, a 2x "im głupszy" stutter at the very start of a video, that neither the primary transcript nor `verify.py`'s acoustic cross-check had caught at all. The tradeoff: CTC's output is phonetically rough (read it for repeat *structure* only, never for wording), and it needs its own repeat-shape detection tuned differently from Whisper's -- see `ctc_check.py`'s two detectors (n-gram-anywhere vs. immediately-adjacent) below.

## Setup (one-time, per machine)

Needs: ffmpeg on PATH, and a Python environment with `faster-whisper`, `numpy`, `scipy`, `transformers`, and CPU-only `torch`. `transformers` pulls in `ctc_check.py`'s wav2vec2 model (`jonatasgrosman/wav2vec2-large-xlsr-53-polish`, ~1.2GB, downloaded automatically on first use and cached by `huggingface_hub`).

**macOS**
```
brew install ffmpeg
python3 -m venv venv-whisper
source venv-whisper/bin/activate
pip install torch --index-url https://download.pytorch.org/whl/cpu
pip install -r requirements.txt
```

**Windows**
```
winget install Gyan.FFmpeg   # restart the shell afterward so ffmpeg is on PATH
python -m venv venv-whisper
venv-whisper\Scripts\activate
pip install torch --index-url https://download.pytorch.org/whl/cpu
pip install -r requirements.txt
```

No WSL, no Smart App Control workarounds needed -- this skill deliberately uses `faster_whisper` directly (not the full `whisperx` package) specifically because that avoids a pyannote-audio/`av` dependency that can trip Windows' Smart App Control. Should run unmodified on Linux too (same install pattern as macOS).

Run every script in this skill through that venv's Python (e.g. `venv-whisper/bin/python scripts/chunk_transcribe.py ...` on macOS/Linux, `venv-whisper\Scripts\python.exe scripts\chunk_transcribe.py ...` on Windows).

## Pipeline

```
projects/<name>/
  raw.mp4                 <- input
  transcript_a.json       <- chunk_transcribe.py output (primary)
  transcript_b.json       <- transcribe2.py output, full-file (on-demand only, see step 2)
  cutlist.json            <- editorial decisions, written by Claude reading the transcript(s)
  segments/                <- cached rendered segments (hash-named) + cached audio envelope
  output.mp4              <- rendered result
```

### 1. Set up the project

```
mkdir -p projects/<name>/segments
cp <source video> projects/<name>/raw.mp4
```

### 2. Transcribe

Both scripts below support an optional Groq API backend (`TRANSCRIBE_BACKEND=groq`,
same `whisper-large-v3` model, seconds instead of minutes, audio leaves the machine)
alongside the default local `faster-whisper` path -- see the repo [README](../../../README.md#transcription-backend-local-vs-groq)
for setup, cost, and the local/cloud tradeoff. Falls back to local automatically if
`GROQ_API_KEY` is missing or a Groq call fails.

Primary pass (chunked, splits at real measured pauses, independent decode per chunk):

```
python scripts/chunk_transcribe.py projects/<name>/raw.mp4 projects/<name>/transcript_a.json --language pl --model large-v3
```

**`transcribe2.py` (single full-file decode) is no longer a default step -- confirmed user preference.** It used to run automatically as a cross-check pass. It is now on-demand only: run it for a specific span if something in the primary transcript still reads ambiguously after cutlist review, not as a blanket second pass over the whole file every time. Two reasons this changed: (1) explicit user feedback that short working increments plus a fast review/feedback loop beat one slow, exhaustive upfront pass; (2) the acoustic tools (`bursts.py`/`find_repeats.py`, see step 3 and `verify.py` in step 5) now cover most of what the cross-check pass was for -- catching a repeat `chunk_transcribe.py` missed -- without needing a second full transcription:
```
python scripts/transcribe2.py projects/<name>/raw.mp4 projects/<name>/transcript_b.json --language pl --model large-v3
```

Always use `large-v3`. `medium` is faster but has been directly observed to silently drop repeats that `large-v3` catches on identical audio -- it is not safe for a final editorial decision, only for a quick throwaway preview if speed is more important than correctness.

**If using the Groq backend (see below), lean on the acoustic tools more, not less, for repeat detection specifically.** Confirmed on real footage: `transcribe2.py`'s whole-file single-pass decode suppressed a known 3x repeat down to one sentence via Groq, on a clip where the same script/same audio via local `faster-whisper` correctly preserved all 3 as separate segments -- Groq's hosted API doesn't expose the decode-level knobs (`condition_on_previous_text`, `compression_ratio_threshold`, etc.) the local path already tunes to resist this. The good news: `chunk_transcribe.py` (the primary pass, pause-chunked) preserved the same 3 repeats correctly via Groq too, since each chunk is decoded independently with much less context for the suppression heuristic to act on -- so the primary-pass workflow is unaffected, but don't expect `transcribe2.py` to add the same protection via Groq that it does locally.

Both scripts run natively on Windows via `faster_whisper` directly (not the full `whisperx` package) specifically to avoid a Smart App Control DLL block on `pyannote-audio`'s `av` dependency. No WSL required.

### 3. Build the cutlist -- Claude reads the transcript(s), not a script

This is deliberately not automated. Read the transcript text and decide what to keep. Rules that held up across a full development session:

- **Keep-last-take rule**: when a phrase/clause is restarted, keep only the final instance that *continues into new content*. An earlier restart can sound perfectly complete in isolation and still be the wrong one to keep -- check whether a later instance restates the same opening and continues further; if so, the earlier one is a discarded take, not a separate thought.
- **Cross-reference when something looks off -- but not by re-transcribing a short isolated clip.** If a number, name, or clause reads strangely, check the other transcript pass first. If you still need to re-check a specific span, prefer `scripts/bursts.py` + `scripts/find_repeats.py` (acoustic, cannot hallucinate) over re-transcribing a tight isolated cut of that span (confirmed to hallucinate matching-but-wrong text when given too little context -- see "Why this shape" above). If you must re-transcribe a span in isolation, use a wide window (10s+) with natural surrounding context, never a ~1s crop.
- **Anomalously long single-word timestamp = red flag.** A short word (e.g. "pomija.", "hajtiketowe") reported with a >1s duration usually means a repeated utterance got folded into that word's span instead of being transcribed. Confirm with `scripts/bursts.py --start <t-2> --end <t+3>` on the raw file -- a >1.2s "word" that's actually 2-4 separate acoustic bursts is the same signature `verify.py` looks for post-render (see step 5); catching it before the cutlist is written is cheaper than catching it after.
- **When unsure whether something is a genuine stutter/retake, run `scripts/find_repeats.py` on the raw file for that span before deciding.** It DTW-matches acoustic bursts against each other independent of what Whisper transcribed, so it catches exactly the cases repeat-suppression hides from the transcript. Calibrated on real footage: genuine repeats landed at DTW distance 0.085-0.24, and the first false match between two different (but similarly-structured, same-speaker) sentences was 0.25+ -- so a low distance is strong evidence, but read it as *ranked* evidence and cross-check the transcript context, not a hard yes/no. It also runs a **deep scan** by default: any burst notably longer than its neighbors gets treated as a haystack, and short bursts nearby get tried against it as subsequence-DTW queries (`local_alignment.py`) -- this specifically catches a retake whose final successful attempt is acoustically fused (no full sustained-silence gap) with the different content that follows it, which whole-burst comparison alone cannot see. Validated on real footage: this independently rediscovered a hidden 3rd retake attempt that the original manual edit had needed an isolated re-transcription to find. It is not a complete fix -- it only works when there's *some* separate short burst nearby to use as a query; if every attempt in a cluster is fused together with zero separation at all, `bursts.py`'s duration pattern (several short bursts, then one much longer one) is still the fallback tell.
- **Run `scripts/ctc_check.py` on the raw file before finalizing the cutlist -- it's a full second opinion, not a targeted spot-check.** Unlike `find_repeats.py` (acoustic, needs a burst boundary somewhere) or the anomalous-word-timestamp check (needs Whisper to at least bloat a word), `ctc_check.py` runs an independent wav2vec2 CTC pass that has no repeat-suppression bias at all and needs neither trigger -- it catches the case that slips every other net: a short, genuinely gapless repeat with no bloated word and no acoustic boundary to anchor a query on. Pass `--whisper-transcript transcript_a.json` to auto-flag which CTC-found repeats the primary transcript already shows (lower priority) vs. which it doesn't (`**NOT in Whisper transcript**` -- highest priority, read every one of these). Confirmed on real footage: caught a 2x "im głupszy" stutter at the very start of a video that neither the transcript nor `verify.py`'s acoustic check had found. CTC's own output is phonetically rough -- read it only for repeat *structure* (this n-gram/phrase appeared twice), never for wording, and always confirm a flagged span against the raw audio or `bursts.py` before cutting.
- **Stock CTA/outro phrases mark a natural ending.** A phrase like "follow for more [value]" is a strong signal the speaker is wrapping up. If it appears more than once, that usually means the speaker re-recorded the whole ending with more material -- ask the user whether they want the first (shorter) or last (more complete) ending; do not assume "more complete = better" without checking, since the user may prefer the punchier first take.
- **Flag, don't silently decide, when a retake changes the actual content** (e.g. different numbers/stats between takes) -- that's an editorial call belonging to the user, not a filler-word judgment call. If the user says to be autonomous, apply the standard last-take rule and document the decision in the cutlist note so it's a one-line edit to reverse.
- Every `keep` entry's `note` field should say *why*, and which transcript source was trusted if there was a discrepancy. Cutlist notes are the debugging trail for the next review pass.

`cutlist.json` schema:
```json
{
  "source": "raw.mp4",
  "pad": 0,
  "keep": [
    {"start": 7.9, "end": 11.13, "note": "why this range, why not the earlier attempt, which transcript source"}
  ]
}
```
`pad` is a legacy fallback; leave at `0` -- boundary padding is handled by the snapping logic in `render.py`, not a fixed offset.

### 4. Render

```
python scripts/render.py projects/<name> [--out output.mp4]
```

For each kept clip: snaps `start`/`end` to the true voice boundary measured from the audio's own energy envelope (not the ASR timestamp), hashes the snapped range, re-encodes only on a cache miss, then concatenates. Editing one cut in `cutlist.json` and re-rendering only re-encodes that one segment -- this is the "10x faster" iteration loop.

Boundary snapping requires a *sustained* silence (>=150ms) before accepting it as a real boundary, specifically to avoid stopping on a brief stop-consonant or breath mid-phrase (this truncated a real word in early testing before the fix).

**Gotcha: nudging a `start`/`end` value by a small amount can silently do nothing.** `snap_start`/`snap_end` (in `audio_boundaries.py`) search a fixed tolerance window around the value you give (`start`: -0.3s to +1.0s; `end`: -2.0s to +0.3s) and grab the *first* real voice-boundary edge they find in it -- not the edge closest to your new value. If the edge you're trying to move away from is still inside that window, the snapped result -- and therefore the segment-cache hash -- comes out identical, `render.py` silently reuses the old cached segment, and your edit had zero effect even though the command reported success. Confirmed on real footage: moving a `start` from 109.6 to 110.0 produced byte-identical output both times. If a boundary fix doesn't seem to take effect after re-rendering, don't just nudge the value further and hope -- query the actual candidate edges directly:
```
python scripts/bursts.py projects/<name>/raw.mp4 --start <t-1> --end <t+1>
```
and pick a new `start`/`end` value that pushes the *old* edge outside the tolerance window (e.g. for `start`, the new value minus 0.3s must be past the old edge) so the snap is forced onto the next real one.

### 5. Verify -- mandatory, not optional

```
python scripts/verify.py projects/<name>/output.mp4
```

Re-transcribes the actual rendered file with `large-v3` (independent of whatever built the cutlist) and reports: repeated word n-grams, anomalously long single-word timestamps, and silence gaps over 0.4s. This is a "narrow down where to listen" tool, not a pass/fail gate -- it will flag genuine parallel rhetoric as a false positive (verified: it correctly flagged three real repeated phrases in a clean approved cut, all of which turned out to be intentional parallel sentence structure, not missed duplicates).

**Supports `TRANSCRIBE_BACKEND=groq`, same as the transcription scripts (falls back to local automatically).** This used to be hardcoded to local `faster-whisper` well after Groq was added everywhere else in the pipeline, which made it the dominant cost of the entire rough-cut + captions workflow by a wide margin -- measured directly: 4 minutes local vs. 9.6 seconds via Groq on the same 70s rendered clip (~25x). Spot-checked on a video with fully known ground truth after the change: identical silence-gap findings (that check doesn't depend on the transcription backend at all), zero repeats missed, and the same "bloated word, acoustically dismissed" noise pattern as the local run (different specific words flagged, since Groq's own transcription has its own minor artifacts, but the acoustic cross-check still correctly waves them off). Not exhaustively re-validated across many videos yet -- watch the next few real runs a little more closely than usual.

**Every text-level flag is now cross-checked acoustically** (via `bursts.py` + `find_repeats.py`'s DTW comparison, run automatically unless `--no-acoustic-check` is passed) and labeled:
- `[CONFIRMED]` -- a close acoustic match exists nearby (either a direct whole-burst DTW pair, or a deep-scan subsequence match into a fused burst); treat as high-priority, this is real audio, not just repeated text. Note this confirms the *acoustic* question (was this really said twice), not the *editorial* one (mistake vs. intentional device) -- confirmed on real footage: a genuine `[CONFIRMED]` case turned out to be deliberate parallel rhetoric ("your ads will work if you don't have X... but will work much better if you have X"), correctly identified as a real repeat, still requiring a human/Claude judgment on whether it's a keeper.
- `[NO_MATCH]` -- no close acoustic match nearby, including after a deep-scan attempt; more likely a transcription artifact than a real duplicate. Confirmed necessary on real footage: a Whisper decoder repetition-loop produced 10 identical-looking text-repeat flags for a single clean sentence, and the acoustic check would have correctly dismissed all 10 (the audio waveform at that point showed no anomaly at all). Also confirmed correctly dismissing a genuine case of the same *term* (not the same *moment*) reused across unrelated sentences -- e.g. a recurring acronym central to the video's topic -- since that's expected vocabulary reuse, not a duplicated take.
- `[INCONCLUSIVE]` -- no long burst nearby to even attempt a deep scan against. This should now be rare (the deep-scan fallback handles most fused-burst cases); when it does happen, a human ear still matters most.

Even a `[NO_MATCH]` or a clean report is not proof of anything by itself -- see "Why this shape" above on why neither ASR nor acoustic matching alone is ground truth. Read every flag; do not treat a clean report as a substitute for actually watching the video, and never let any of these automated signals override a specific problem the user reports.

**`verify.py` also runs `ctc_check.py`'s CTC second-opinion pass on the rendered output itself** (skip with `--no-ctc-check` for a faster text-only preview). This closes a real gap, not a hypothetical one: on one project, a render that had already passed the Whisper-transcript + acoustic checks above still had a hidden 4x fused repeat and, later in the same video, a separate 4x fused "ale" stutter -- neither had a bloated word timestamp or an acoustic burst pair below the distance threshold, so nothing above caught either one. Both were only found because a human reported the video still sounded wrong, which prompted running CTC manually on the flagged span. Since CTC has no repeat-suppression bias and needs neither trigger to work, running it as part of `verify.py` itself catches this category automatically instead of depending on a human noticing first. Findings are tagged `[ALSO in Whisper transcript]` (lower priority, already visible another way) or `[**NOT in Whisper transcript -- highest priority**]` (the exact shape every other check in this file was built to close and still occasionally misses) -- read every CTC finding for repeat *structure* only, never wording (same caveat as `ctc_check.py` itself).

## Iterating after review

**The stable, repeatable loop this skill converged on, after several rejected alternatives (see below) -- four phases, each with a specific job:**

1. **Automatic first cut (Claude's job, no human input yet).** Transcribe, run `ctc_check.py` on the *raw* file before writing the cutlist (not just spot-checking suspicious spans), build the cutlist, render, then run `verify.py` -- which as of this section now includes its own CTC pass on the *rendered output* (see below), not just Whisper + acoustic cross-checking. Iterate this phase entirely on your own until `verify.py` comes back clean or every flag is understood and dismissed with a reason. The goal is that phase 2 has as little left to catch as possible -- every automated check that exists should already have run before a human ever watches the file.
2. **Human tagging (fast, imprecise is fine).** The user watches the delivered file (or `review.html`) and marks what's still wrong -- plain language ("still sounds off around 0:43") or a precise-ish in/out range via the review console's `I`/`O` shortcut. **Their coordinate does not need to be exact** -- that's phase 3's job, not theirs. Don't ask them to be more precise; that just moves the overhead onto the person least equipped to use the acoustic tools.
3. **Precise fix (Claude's job) -- never trust the human's coordinate as the cut point.** Run `bursts.py` and, if the transcript doesn't already explain it, a targeted `ctc_backend.transcribe_words(start=..., end=...)` call around their flagged region *before* editing `cutlist.json`. This is not optional: confirmed directly on real footage that applying a user's flagged output-time range literally, without checking for a real silence gap nearby, produced a *worse* cut than the one being fixed (landed mid-stutter instead of at the actual pause a few seconds away). Once the real boundary or repeat structure is found, edit `cutlist.json` and re-render. **Do not re-run `verify.py` or regenerate captions after every single micro-fix** -- that full-pipeline cost is what makes iteration feel slower than doing the edit by hand. Re-render only (cache-fast, a few seconds), let the human spot-check that one change, and batch the full `verify.py` + caption regeneration + redeliver for once at the end, when the user says they're done.
4. **Close the gap, don't just patch the video (Claude's job, after phase 3).** If a human caught something in phase 2 that phase 1's automated checks should have caught, that's a permanent hole in the tooling, not a one-off miss -- fix the *tool*, not just this video. Concretely: `verify.py` was missing a CTC pass on the rendered output for a long time (CTC only ran raw-file-build-time, per step 1 above) -- two separate real misses in one project (a 4x fused repeat, then later a 4x fused "ale" stutter) both would have been caught automatically if `verify.py` itself ran CTC, instead of requiring a human to notice the video still sounded wrong first. That gap is now closed (see `verify.py`'s CTC section below) -- the next miss like it should become a new automated check the same way, not another round of "watch closer next time."

**Two things tried and rejected before landing on the loop above, worth knowing so they aren't re-attempted the same way:**
- **An in-browser drag-to-trim editor** (edit `cutlist.json` start/end directly by dragging timeline handles, re-render on release) was built, tested hands-on, and rejected: "super counterintuitive, hard to work on, no undos, ctrl+z doesn't work, can't split a chunk into smaller ones... didn't make the work easier." A trim UI without undo/redo and split *is* a video editor, not a small addition to a review page -- don't re-attempt in-browser cutlist editing without solving those first, that's the real scope. `review_console.py`'s flag list plus the `I`/`O` in/out range shortcut (still flag-only, no write, no re-render from the browser) is what replaced it.
- **Applying a human's flagged coordinate literally, as the exact cut point.** Confirmed to actively make a cut worse on real footage (see phase 3 above) -- the fix is always to re-derive the precise boundary from the acoustic tools, using the human's coordinate only to know *where to look*, never as the value to write into `cutlist.json`.

For manual single-clip edits outside the phase 2/3 loop above (e.g. a raw-footage timestamp read directly off a transcript):

- **If given a raw-footage timestamp** (rare -- usually only if looking at the transcript), edit the matching `cutlist.json` entry directly.
- **If given an output-relative timestamp** ("the pause at 0:45 is too long", "cut more around 1:12 in the video"), that time is in the *edited* video, not the raw source -- `cutlist.json` only understands raw-footage time. Run:
  ```
  python scripts/locate.py projects/<name> <output_time_seconds>
  ```
  This recomputes the same snapped boundaries `render.py` uses (so it matches what's actually in `output.mp4`, not the unsnapped cutlist values) and reports which `keep` entry the time falls in, plus the exact corresponding raw-footage time to edit.
- After editing, re-run `render.py`. A manual single-clip edit can introduce a new issue (e.g. a boundary that now clips a word) just as easily as the original cutlist construction could -- but per phase 3 above, save the full `verify.py` re-run for the end of the human-iteration loop, not after every single edit.
- **`render.py`'s boundary snap grabs the *first* edge within its tolerance window, not the one closest to your intended value** -- confirmed directly costing a wasted round-trip: moving a clip's start from 113.4 to 113.95 to skip a fused "coś, coś" stutter still included both, because the snap found an unrelated edge at 113.65 sitting inside the tolerance window. Fix is the same as documented in the render section below: push the requested value far enough that the window no longer reaches the unwanted edge, then re-verify (re-transcribe just that segment, or re-run `verify.py`/`ctc_check`) to confirm the fix actually took effect -- don't assume it worked just because `render.py` didn't error.

**Give the user two review aids instead of just `output.mp4`, so their feedback comes back as coordinates instead of prose to be guessed at.** A guessed "cut too early at ~12s" cost a wrong fix and a redo in this project's real usage -- eliminate the guess, don't get better at it.
```
python scripts/annotate_cuts.py projects/<name>          # -> output_review.mp4 + a numbered cut-map (clip N: output range, raw range, note)
python scripts/boundary_reel.py projects/<name>          # -> boundary_reel.mp4: ±1.5s around every internal cut join, concatenated
```
- `annotate_cuts.py` burns a running timecode + "clip N" label onto a throwaway review copy and prints the same numbering as a cut-map. Once the user has this, feedback can be "clip 4, starts mid-word" instead of a timestamp -- there's no coordinate left to get wrong in translation, in either direction.
- `boundary_reel.py` extracts a short window around every internal join (using `timeline.json`'s cut points, the same ones `locate.py` already understands) and concatenates them with a join-number label. Almost every real problem found in this project lived at a join, not in the middle of an untouched clip -- a full re-watch spends most of its time re-confirming footage that didn't change. Point the user at this first; fall back to the full annotated video only if something might be wrong mid-clip.
- Both write clearly-labeled throwaway files (`output_review.mp4`, `boundary_reel.mp4`) -- never confuse these with the deliverable, and don't let a later burn/copy step accidentally overwrite the real `output_captioned.mp4` with one of these (confirmed this is an easy mistake to make: reusing a scratch/temp `captions.json` for an unrelated test once silently clobbered the properly-styled project file -- always point review/test tooling at scratch paths, never at a live project's delivery file, even for "just a quick check").

**A full review console (`scripts/review_console.py`) goes one step further: an actual page with the video, a clickable timeline, and a flag button so the user marks problems while watching instead of writing prose at all.**
```
python scripts/review_console.py projects/<name>          # -> review.html, embeds output_review.mp4 by default
```
Serve it with the `review-console` config in `.claude/launch.json` (`range_server.py`, see below -- not `python -m http.server`) and open `http://localhost:<port>/<project_name>/review.html`. The user watches, clicks a flag type (pauses the video, captures the exact time), optionally adds a short note, and hits "copy flags for chat" to paste a ready-made list back -- e.g. `- 0:18.2 (clip 3) [repeat]: upselle said twice`. **Deliberately does not support trimming/editing/re-rendering from the page** -- that would mean rebuilding a video editor and abandoning `cutlist.json` as the single source of truth that makes the render/verify loop fast and auditable. It only converts what the user hears into a coordinate; editing stays with Claude.

**A drag-to-trim version of this (edit start/end directly in the browser, write straight to `cutlist.json`, re-render on release) was built, tested, and reverted in the same session.** It worked mechanically, but confirmed user feedback after hands-on testing: "super counterintuitive, hard to work on, no undos, ctrl+z doesn't work, can't split a chunk into smaller ones... didn't make the work easier." The scope note above ("would mean rebuilding a video editor") wasn't a hypothetical -- it's exactly what happened: a real trim UI needs undo/redo and split, at which point it *is* a video editor, and a half-built one is worse than the flag-only version it replaced. Don't re-attempt in-browser cutlist editing without solving undo/redo and split first; that's the actual scope, not a nice-to-have on top.

**Instead, press `I` at the start of a bad stretch and `O` at the end (or use the "mark start"/"mark end" buttons) to flag a precise in/out range, not just a single point.** This is still flag-only -- no write to `cutlist.json`, no re-render -- but gives Claude both boundaries of what to cut directly, e.g. `- 0:12.3–0:15.1 [cut-range]: repeated phrase`, closer to the precision the trim UI was trying to provide without any of its editing-model problems (undo is just "don't press O", nothing was written until you did).

Two real bugs were found and fixed building this, both worth knowing if this ever needs touching again:
- **Python's stdlib `http.server` has no Range-request support at all** (confirmed by reading `SimpleHTTPRequestHandler.send_head`'s source directly, even on Python 3.13) -- every request came back a full `200 OK`, never `206 Partial Content`, so a `<video>` element could not seek without re-downloading the whole file, and `video.currentTime = X` silently reset back to `0` instead of seeking. `scripts/range_server.py` is a small custom static server that actually implements `Range: bytes=...` -- use it (via `.claude/launch.json`), not `python -m http.server`, for anything serving video to a browser.
- **`render_overlay`'s ffmpeg output needs `-movflags +faststart`.** Without it the moov atom sits at the end of the file, so a browser can't even determine duration/seek points without downloading everything first -- this compounded with the Range issue above to make seeking fail outright. `annotate_cuts.py`'s burn step now sets this; any other ffmpeg output meant to be played over HTTP (not just opened locally) needs it too.
- Also: `range_server.py` must use `socketserver.ThreadingTCPServer`, not plain `TCPServer` -- a single-threaded server lets one kept-alive browser connection block every subsequent request forever, which hung the entire page load the first time this was tested.
- And: `navigator.clipboard.writeText` can fail (permission denied, lost focus, older browser) -- confirmed happening in a real test session. The copy button falls back to a visible, pre-selected textarea if the Clipboard API rejects, so a flag list is never trapped behind a silently-failing button.

## Known limitations

- **`scripts/find_repeats.py`'s whole-burst DTW comparison has one blind spot, now mostly closed by a deep-scan pass, not fully.** If a retake's final successful attempt is acoustically fused (no full sustained-silence gap) into the same burst as the different content that follows it, the earlier false-start bursts won't DTW-match well against that one long fused burst -- most of it is different content. `deep_scan_long_bursts` (in `find_repeats.py`, using `local_alignment.subsequence_match`) treats any burst notably longer than its neighbors as a haystack and tries nearby short bursts as subsequence-DTW queries against it, which finds a repeat *inside* the long burst without needing the whole thing to match. Validated on real footage across two different projects: correctly found a hidden repeat inside a 15.5s fused burst that whole-burst DTW couldn't see, and independently rediscovered a hidden 3rd retake attempt that the original manual edit had needed an isolated re-transcription to catch. Remaining gap: this only works when there's some separate short burst nearby to use as a query -- if an entire cluster of repeats has zero separation anywhere (no burst boundary at all to anchor a query on), neither pass can find it, and `bursts.py`'s raw duration pattern is the only remaining signal.
  - A from-scratch Smith-Waterman-style local aligner was tried first for the deep-scan pass and abandoned: with a workable gap penalty it produced a spurious "alignment" spanning nearly an entire 15.5s span at only 0.42 average frame similarity (real repeats measured 0.67-0.75), because it could thread cheap gaps through occasional coincidental high-similarity frames without matching content throughout. Subsequence (open-begin/open-end) DTW was used instead -- no free gap-penalty parameter, reuses the same calibrated per-frame cost as whole-burst DTW -- but has its own failure mode: unconstrained, it can "stall" on a single haystack frame and match many query frames to it almost for free (confirmed: a false match at cost 0.227, inside the calibrated true-repeat range, where the matched window was ~5x shorter than the query). `subsequence_match` rejects any match whose duration ratio to the query exceeds 2.5x; this filter is required, not optional.
- DTW distance (whole-burst or subsequence) is *ranked* evidence, not a clean binary. Same-speaker, similarly-structured-but-different sentences can land close to the calibrated true-repeat range (observed floor: 0.25) -- a low distance is strong support, but cross-check the transcript/context before treating any single number as a verdict. Separately: a low distance confirms the audio really was repeated, not that repeating it was a *mistake* -- deliberate parallel rhetoric acoustically looks exactly like an accidental retake, and that distinction is still an editorial judgment call, not something these tools can make.
- **`find_repeats.py`'s default `--window` is 20s, tightened down from an initial 60s after testing on a full ~168s video** -- a real retake is always spoken within seconds of the original, and the wider window let in same-speaker/same-genre coincidental noise from bursts 50-100+ seconds apart without adding any true positives. If a retake genuinely happens further apart than 20s in some footage, widen `--window` for that specific check.
- No transcription pass, model size, or decode setting fully eliminates repeat-suppression -- and this isn't limited to short/degraded audio; it was confirmed on a full natural-context, pause-chunked pass too, with multiple independent transcription methods agreeing on the same wrong (over-suppressed) answer. Treat every transcript as a lossy hint. Acoustic checks (`bursts.py`, `find_repeats.py`, `verify.py`'s cross-check) are a second, independent line of evidence precisely because they cannot suppress or hallucinate the way a decoder can -- but actually watching the render is still the only complete check.
- Isolating a short (~1s) audio clip to re-transcribe it "for confirmation" is unreliable in both directions -- it can hallucinate matching text for genuinely different audio, and it provides no more protection against repeat-suppression than a full-context pass. Don't use it as evidence either way; use `bursts.py`/`find_repeats.py` instead for acoustic questions, and a wide-context (10s+) re-transcription if a text question specifically needs re-checking.
- **`ctc_check.py`'s output is phonetically rough and needs `--whisper-transcript` to be useful for prioritization** -- without it, every finding looks equally urgent, which isn't true (a repeat Whisper already caught is much lower priority than one it didn't). Cross-referencing is by time-window overlap (3s default), not text match, since CTC's spelling and Whisper's rarely agree closely enough to compare directly.
  - Two acoustic-only alternatives to CTC were tried first and abandoned, both on real footage: (1) within-a-span self-similarity -- a genuine fused repeat's average self-similarity measured *lower* than a non-repeated control, because a real retake changes pace enough that it isn't acoustically identical to itself, and ordinary speech has enough baseline self-similarity (repeated phonemes, prosody) to swamp the thin remaining margin; (2) front-half-vs-back-half subsequence matching within one suspect span -- no fused repeat crossed the calibrated match threshold, confirming that `find_repeats.py`'s insistence on a real burst boundary for its deep-scan is load-bearing, not a limitation to route around. Both findings are why the fix had to be a different decoder architecture, not more DSP cleverness on the same features.
  - CTC runs at roughly 0.6x realtime on CPU (a 135s video costs ~80s) -- affordable as a background pass at cutlist-build time, not something to run repeatedly per small edit the way `verify.py` runs after every render.
  - Model quality is a real, unquantified variable -- `jonatasgrosman/wav2vec2-large-xlsr-53-polish` is a small multilingual-base fine-tune, not evaluated here beyond the specific cases it was validated against. Treat every CTC-only finding (no Whisper corroboration) as "listen to this," not "this is definitely a missed repeat."
- This skill covers rough-cut only (dead air, fillers, retakes). Graphics, captions, and music are separate stages, not built here.

## File manifest

- `scripts/chunk_transcribe.py` -- primary transcription: chunks at measured real pauses, independent decode per chunk, `large-v3` default.
- `scripts/transcribe2.py` -- single full-file transcription pass, on-demand only (not a default step, see step 2) -- for cross-checking a specific span if it still reads ambiguously after the primary pass.
- `scripts/audio_boundaries.py` -- energy envelope + sustained-silence boundary snapping (`VoiceEnvelope.snap_start`/`snap_end`), plus acoustic burst segmentation (`VoiceEnvelope.list_bursts`). Shared by `render.py`, `chunk_transcribe.py`, `bursts.py`, and `find_repeats.py`.
- `scripts/bursts.py` -- CLI: prints/writes the acoustic burst inventory (start/end/duration of every voiced span) for any media file, raw or rendered. ASR-independent; a stutter/retake shows up here as a duration pattern even when Whisper's transcript hides it.
- `scripts/dtw_features.py` -- shared log-mel feature extraction + normalized whole-sequence DTW distance, used by `find_repeats.py`'s whole-burst pass and `verify.py`'s acoustic cross-check.
- `scripts/local_alignment.py` -- subsequence (open-begin/open-end) DTW: finds where a known short burst best re-occurs inside a longer span, without needing the whole span to match. Powers the deep-scan pass in `find_repeats.py` and `verify.py`'s fused-burst fallback.
- `scripts/find_repeats.py` -- ASR-independent repeat detector, two passes: whole-burst DTW (tempo-invariant, unlike the original fixed-time-lag v1) for cleanly-separated retakes, plus a deep scan (`local_alignment.py`) for retakes fused into a longer burst with no silence gap. Calibrated on real footage: see Known limitations for thresholds and remaining blind spot.
- `scripts/render.py` -- cutlist -> boundary-snapped, hash-cached segments -> concat.
- `scripts/verify.py` -- automated post-render sanity check: repeated n-grams, bloated word timestamps, and silence gaps, each cross-checked against `bursts.py`/`find_repeats.py`'s acoustic evidence (including the deep-scan fallback) and labeled CONFIRMED/NO_MATCH/INCONCLUSIVE, plus a CTC second-opinion pass on the rendered output itself (`--no-ctc-check` to skip) that has caught real fused repeats none of the other checks here saw.
- `scripts/locate.py` -- maps an output-relative timestamp back to the raw-footage time and cutlist entry it came from, for iterating on feedback given against the rendered video.
- `scripts/ffmpeg_util.py` -- shared ffmpeg binary resolution (PATH, with a winget-install fallback path).
- `scripts/groq_backend.py` -- optional cloud transcription backend (`TRANSCRIBE_BACKEND=groq`): extracts audio, calls Groq's `whisper-large-v3` API, normalizes the response to this pipeline's segment/word schema. See repo README for setup.
- `scripts/ctc_backend.py` -- second-opinion transcription via wav2vec2 CTC (`jonatasgrosman/wav2vec2-large-xlsr-53-polish`), no repeat-suppression bias by construction (unlike Whisper's autoregressive decoder). Word-level timestamps are reconstructed from CTC's own frame-level output (~20ms/frame) -- approximate, good enough to localize a repeat, not to snap a cut boundary. Output is for repeat-structure detection only, never a transcript source (phonetically rough, small multilingual model).
- `scripts/ctc_check.py` -- CLI: runs `ctc_backend.py` over a raw file, finds repeated phrases two ways (n-gram-anywhere-in-file, for retakes spoken far apart; immediately-adjacent-repeat, for genuinely gapless fused stutters that an n-gram window structurally can't see), and cross-references each against the primary Whisper transcript's own repeats if `--whisper-transcript` is given. This is the tool that closes the gap `find_repeats.py`'s acoustic approach and the bloated-word-timestamp check both leave open: a short, gapless repeat with no bloated word and no burst boundary to anchor a query on.
- `scripts/annotate_cuts.py` -- review aid: burns a running timecode + "clip N" overlay onto a throwaway copy of `output.mp4`, and prints/writes a numbered cut-map (clip N: output range, raw range, note). Lets user feedback reference "clip 4" instead of a guessed timestamp. Requires an explicit `fontfile` for `drawtext` (reuses `../../captions/assets/Montserrat-SemiBold.ttf`) -- confirmed this system's ffmpeg build has no working Fontconfig setup, so `drawtext` without one fails loudly ("Cannot load default config file").
- `scripts/boundary_reel.py` -- review aid: extracts a short window (default ±1.5s) around every internal cut join from `timeline.json` and concatenates them into one short labeled reel. Concentrates review time on cut joins, where almost every real problem in this project's actual usage has lived, instead of a full re-watch that mostly re-confirms unchanged footage.
- `scripts/review_console.py` -- generates `review.html`: video + clickable timeline (clip boundaries from `timeline.json`/`cutlist.json`) + a flag-recording panel that exports a paste-ready coordinate list. See "Give the user two review aids..." above for the two real serving bugs (no Range support, missing `+faststart`) this needed to actually work in a browser.
- `scripts/range_server.py` -- small static file server with actual HTTP Range-request support (threaded), for serving video to `review.html`. Python's stdlib `http.server` has none at all -- use this instead, via `.claude/launch.json`'s `review-console` config.
