# Captions

Rendered video -> word-level transcript of THAT file -> glossary-corrected, grouped caption cards (Claude's editorial pass) -> burned-in captions via ffmpeg/libass, Montserrat SemiBold by default.

## Setup (one-time, per machine)

Needs ffmpeg on PATH and a Python environment with `faster-whisper` (CPU-only `torch`). If you also have the `rough-cut` skill installed, they can share one venv -- just `pip install -r` both skills' `requirements.txt` files into it.

```
brew install ffmpeg          # macOS -- Windows: winget install Gyan.FFmpeg, then restart the shell
python3 -m venv venv-whisper
source venv-whisper/bin/activate   # Windows: venv-whisper\Scripts\activate
pip install torch --index-url https://download.pytorch.org/whl/cpu
pip install -r requirements.txt
```

Fonts and the ffmpeg-lookup fallback are cross-platform (no OS-specific paths in the scripts) -- this should run unmodified on macOS, Windows, or Linux.

## Why transcribe the video again, when rough-cut already has a transcript

The rough-cut transcript is timed against the *raw* source footage; captions need to be timed against the *edited* output, which has a different timeline (clips removed, boundaries snapped). Always transcribe the final video you're captioning, never reuse an upstream transcript.

## Pipeline

```
projects/<name>/
  output.mp4          <- already-edited video (e.g. rough-cut's output)
  timeline.json         <- optional: exact cut points, if output.mp4 came from rough-cut's render.py
  words.json           <- make_captions.py output (word-level, this file's own timeline)
  domain.json           <- content niche + jargon, CONTEXT for judgment (not mechanical)
  glossary.json         <- confirmed exact corrections + highlight colors (mechanical)
  captions.json         <- editorial: card groupings, glossary fixes applied, highlight flags
  captions.ass           <- generated by burn_captions.py, not hand-edited
  output_captioned.mp4  <- final result
```

### 1. Transcribe the edited video

```
python scripts/make_captions.py projects/<name>/output.mp4 projects/<name>/words.json --language pl
```
Uses `large-v3` (same reasoning as the rough-cut skill: smaller models miss things). Flags any word under 0.6 confidence -- check those against the glossary even if they don't look like an obvious mishearing.

Supports `TRANSCRIBE_BACKEND=groq` for a much faster cloud transcription pass (same `whisper-large-v3` model, via Groq's API) -- see the repo [README](../../../README.md#transcription-backend-local-vs-groq) for setup and the local/cloud tradeoff. Falls back to local automatically if `GROQ_API_KEY` is missing or the call fails. Note: Groq's API doesn't return a per-word confidence score, so the under-0.6-confidence flag above never fires on Groq-sourced transcripts -- a low-confidence word from Groq will look identical to a high-confidence one, so read the transcript itself a bit more carefully in that mode.

### 2. Read the domain profile, then build/extend the glossary

`domain.json` is CONTEXT for judgment, not a mechanical file -- read it before reading the transcript, the same way you'd want background on a topic before editing an article about it:
```json
{
  "niche": "Marketing/business coaching content in Polish -- paid ads copywriting, CTAs, sales funnels...",
  "vocabulary": ["ROAS", "CTA", "call to action", "CPA", "funnel", "coaching", "mentoring", ...],
  "notes": "Numbers followed by 'x' almost always refer to a named ad-performance metric (ROAS)..."
}
```
Copy `domain.example.json` into a project as `domain.json` and fill it in once per speaker/niche -- it's reusable across that speaker's videos, not a one-off.

`glossary.json` is the separate, mechanical file -- **confirmed** exact corrections only, auto-applied by `group_cards.py`:
```json
{
  "terms": [
    {"canonical": "call to action", "mishearings": ["Coldplay Action", "Colt Auction"], "highlight": "#FE2C55"},
    {"canonical": "5x ROAS", "mishearings": ["5x0x", "5 razy"], "highlight": "#25F4EE"}
  ]
}
```
`highlight` is optional; omit it for a silent correction with no color.

**The critical rule, learned the hard way: never add an entry to glossary.json from a guess, however contextually plausible -- confirm with the user first.** "5x ROAS" was first mis-transcribed as "5x0x". My own attempt to fix it, without asking, landed on "5 razy" -- a plausible-sounding but wrong guess ("razy" alone doesn't say what increased 5-fold; ROAS is the standard term for exactly this speaker's content, which domain.json would have flagged as the obvious candidate). Worse: **every re-transcription pass agreed with "razy" too**, because the audio genuinely sounds like that -- cross-checking multiple ASR passes against each other only catches *transcription* disagreement, not a mishearing every pass makes identically. That means low-confidence-score flags (see step 2.5) and multi-pass cross-checks cannot substitute for asking; they narrow down *where* to ask, not *what the answer is*.

**Before finalizing glossary.json, ask the user about every case where:**
- A number, acronym, or name from `domain.json`'s vocabulary plausibly belongs in a sentence, but the transcript reads vague, ungrammatical, or semantically thin instead (this was the exact shape of the ROAS miss -- "5 times" said with no stated subject should read as suspicious on its own).
- A word scored low ASR confidence (see `scripts/flag_ambiguous.py`, step 2.5) near a domain-relevant topic.
- Two transcription passes disagree on a word or phrase.

Ask these as a single batched question set, not one at a time -- respect the user's time and don't waste tokens re-deriving an answer only the user has.

### 2.5. Flag low-confidence words

```
python scripts/flag_ambiguous.py projects/<name>/words.json --min-score 0.6
```
Surfaces words the ASR itself was unsure about. This is the one mechanically reliable signal here -- **an earlier version of this script also tried fuzzy text-matching low-confidence words against domain.json's vocabulary, and that was removed after calibrating it against real cases: genuine mishearings scored *lower* on edit-distance than coincidental false positives** ("0az" vs "ROAS" scored 0.29; "tak" [yes] vs "CTA" scored 0.67 by pure coincidence of both being short strings). ASR mishearings are phonetic, not orthographic, and character-similarity is the wrong tool for that -- no threshold fixes it, because real and noise cases overlap in score. Use judgment informed by `domain.json`, not string-distance, to decide which flagged words matter.

### 3. Generate the draft cards

```
python scripts/group_cards.py projects/<name>/words.json projects/<name>/glossary.json projects/<name>/captions.json --max-words 2 --timeline projects/<name>/timeline.json
```
Mechanically applies glossary substitutions (exact phrase match, case/punctuation-insensitive) and groups words into cards, breaking preferentially at punctuation. This is a **draft** -- read it before rendering:
- Glossary matching is exact-phrase only. It will miss a mishearing you haven't listed yet. Anything that looks like a mangled name, acronym, or number is worth a manual look even if the script didn't flag it (this is the same "don't trust the transcript" discipline as rough-cut, applied to word choice instead of cutting).
- Default is `--max-words 2` -- keeps lines short so text never stretches across most of the screen width, which happened at higher values (3-4) especially with longer words. Adjust up if cards feel too choppy for a given video, but check a wide-word moment (long compound words, or a highlighted multi-word glossary term) before committing to a higher default.
- Each card's `words` list can have a per-word `highlight` (hex color or `null`); this is what group_cards.py sets automatically for glossary matches, but any word can be flagged this way. Pass `--no-highlight` to apply glossary text corrections without the color -- everything renders in the default text color (confirmed preference: plain white reads cleaner than color-coding glossary terms for this style).
- `burn_captions.py` lowercases every word by default (`style.lowercase`, confirmed preference -- inconsistent capitalization, e.g. a word starting a sentence, looked out of place next to the deliberately casual/consistent caption style). Set `"lowercase": false` in a project's `captions.json` style block to preserve original casing.
- **`--timeline` is important whenever `output.mp4` has hard cuts** (e.g. it's rough-cut's output): pass `timeline.json` (written by rough-cut's `render.py` alongside `output.mp4`) so a caption card never spans two different shots. Without it, a card can straddle a cut -- text carries over from a shot that already ended, or a word from the new shot appears without its card really having started, which reads as the caption dying mid-sentence exactly when the scene changes. Note that a single word's own ASR timestamp can itself straddle a cut (the cut is exact, from real editing; the word's timing is only ever approximate) -- `group_cards.py` clamps a word's start to the cut in that case rather than trusting the word's raw timestamp, verified against a real case where this exact thing happened.

### 4. Render

```
python scripts/burn_captions.py projects/<name>/captions.json projects/<name>/output.mp4
```
Builds an `.ass` subtitle file (Montserrat SemiBold, soft gray shadow, bottom-center by default) and burns it in with ffmpeg/libass. Per-word highlight colors are inline ASS override tags within a single caption line, not a second style.

Style overrides go in `captions.json` under a `"style"` key, e.g.:
```json
{
  "style": {"font": "Inter Bold", "font_file": "Inter-Bold.ttf", "font_size": 40, "margin_v": 259},
  "cards": [...]
}
```
See `DEFAULT_STYLE` in `scripts/burn_captions.py` for every overridable field. Notable ones beyond the obvious (size/color/margin):
- **`font_file`** -- lets a specific project point at a different font file than the shared default without changing `DEFAULT_FONT`/`DEFAULT_FONT_FILE` (e.g. a locally-licensed commercial font -- see Font section below).
- **`outline_color`** -- shared between the outline stroke *and* the shadow (they're the same ASS color field). If the shadow looks too dark, lighten this rather than looking for a separate "shadow color" setting that doesn't exist. `#666666` (soft gray) reads as a subtle shadow; `#000000` reads as a hard dark ring especially at low blur.
- **`blur`** and **`shadow`** are independent controls, don't assume one implies the other: `shadow` sets how far/dark the effect reaches, `blur` sets how soft the edge is. Low blur + any shadow value reads as a hard-edged ring (ASS's native shadow is hard-edged by default); high blur (~2.5-3) diffuses the same shadow into a soft, roughly symmetric glow. Tune shadow down for subtlety, keep blur high to avoid it looking like an outline.
- **`letter_spacing`** -- ASS `\fsp` override, in pixels, supports negative values for tighter tracking. CapCut/Canva-style "premium caption" presets commonly apply negative tracking (e.g. CapCut's "character: -1", Canva's "Letter Spacing: -50") -- there's no exact numeric conversion between platforms or into ASS's pixel-based `\fsp`, calibrate visually (around -1.5 at font_size 40 read close in testing).

**Verify styling changes with a tight crop, not the full frame.** At normal video-frame scale, a genuine premium font and a generic system-font fallback can look similar enough to miss the difference -- this is exactly what happened with the font-resolution bug below, caught only after cropping and upscaling just the caption text region. When tuning font/shadow/spacing, always: render, screenshot, crop tight around the text (`ffmpeg -vf "crop=W:H:X:Y,scale=3*W:3*H"`), and compare at that magnification before judging a match.

## Font resolution can fail silently -- always verify

**A font can be wrong in the rendered output with zero error message.** This happened for a full styling session: `fontsdir` (pointing ffmpeg/libass at `assets/` for fonts not installed system-wide) worked correctly for every font built in this repo via fontTools (TikTok Sans, Montserrat, Inter -- all instanced with `--update-name-table`, which bakes the full descriptive name into the font's *family* record), but silently failed for a raw third-party OTF file and fell back to system Arial -- no warning, no crash, just the wrong font rendered at whatever size/weight was requested. It took a tight pixel-level crop comparison to even notice, several styling iterations in.

**Root cause, as far as diagnosed:** the fonts that worked all have their ASS `Fontname` referencing a family name I set myself to be the *exact, full* descriptive string (e.g. "TikTok Sans 36pt ExtraBold" *is* that font's family name, nameID 1, because the instancer was told to make it so). The font that failed had a normal family/subfamily split (family "Proxima Nova", subfamily "Semibold") -- referencing "Proxima Nova Semibold" as `Fontname` doesn't match either field on its own, but even correcting to just "Proxima Nova" (the real family name) still failed, while a control test proved `fontsdir` itself works (TikTok Sans resolved correctly in the same test). That points to the specific file itself, not the family-name mismatch, as autoritative -- possibly connected to it being format-different (OTF/CFF vs the TTF/glyf files built here) or structurally non-standard, consistent with it having come from an unverified source in the first place.

**How to verify a font is actually being used, every time one is added or changed:**
1. Build a minimal test `.ass` with two styles: the real font name, and a deliberately fake one (e.g. `ThisFontDoesNotExistXYZ`).
2. Render one frame with both, piping ffmpeg's stderr through: `ffmpeg -f lavfi -i color=... -vf "ass='test.ass':fontsdir='assets'" ... 2>&1 | grep fontselect`
3. Check the `fontselect: (name, weight, italic) -> resolved` lines. If the real font and the fake one resolve to the *same* fallback name (commonly `ArialMT`/`Arial-BoldMT` on Windows), the real font failed to load -- fix it before trusting any render, styling decision, or screenshot comparison built on top of it.

This is the same underlying discipline as rough-cut's "never trust a transcript, verify the actual rendered file" -- extended to fonts: never trust that a font name in a style block is actually what's rendering, verify it resolved.

## Font

Default is `assets/Montserrat-SemiBold.ttf` (`DEFAULT_FONT`/`DEFAULT_FONT_FILE` in `burn_captions.py`), a static instance built from Google Fonts' variable "Montserrat" (OFL-licensed, see `assets/Montserrat-OFL.txt`) at weight 600 -- confirmed via the fontselect check above to load correctly, and closest visual match found so far to a "premium capcut-style" reference once actually rendering as itself (not a silent fallback). `assets/TikTokSans-ExtraBold.ttf` and `assets/Inter-Bold.ttf` (both OFL) are available alternatives -- just repoint the two constants, and re-verify with the fontselect check since a different file may behave differently.

All are instantiated as static fonts rather than shipped as their variable originals, because ffmpeg/libass's variable-font support is inconsistent across builds; a static instance renders identically everywhere. To build a different weight from a Google Fonts variable font:
```
python -m fontTools.varLib.instancer <family>-Variable.ttf wght=<weight> --update-name-table -o assets/<Family>-<Weight>.ttf
```
(pass `wdth=100 opsz=36 slnt=0` too if the font has those axes, e.g. TikTok Sans does, Montserrat/Inter don't -- check named instances first with fontTools if unsure). The variable source isn't kept in this repo after instancing -- re-download from Google Fonts' GitHub repo (`github.com/google/fonts`, under `ofl/<family>/`) if you need a different instance. Then update `DEFAULT_FONT`/`DEFAULT_FONT_FILE` in `burn_captions.py`, or set `font`/`font_file` in a specific project's `captions.json` style block instead. **Always re-run the fontselect verification above after adding any new font file, especially anything not built by this repo's own fontTools process.**

**On using a commercial font instead (e.g. Proxima Nova):** only do this with your own valid license, and only for your own local use -- do not commit or share the font file with anyone else, including teammates, since that's redistribution regardless of intent. Drop your licensed file into `assets/`, override `font`/`font_file` locally via a project's `captions.json` `style` key (keeps the shared `DEFAULT_FONT` safe for anyone else using the skill), and keep that specific project file out of anything you share or commit. Verify it actually resolves before trusting the result -- a commercial font from an unverified source may fail to load at all, silently, as happened here.

## Vertical position

Captions are positioned with a single fixed `margin_v` for the whole video, not tracked per-shot. A per-shot chin-tracking version was built and then deliberately reverted: it looked more precise on paper, but jumping the caption position between shots read as visually chaotic in practice, and it was real ongoing cost (a frame-by-face survey on every new video). Cheaper and calmer:

1. Pick 3 shots spread across the video (start, middle, end).
2. Extract one representative frame from each (e.g. via `ffmpeg -ss <t> -frames:v 1`), look at each, and estimate the chin's y-position in pixels.
3. Compute `margin_v = frame_height - chin_y - small_gap` (small_gap ~20-30px) for each of the 3, take the **median**, and set that as `style.margin_v`.

This won't be pixel-perfect on shots with unusually tight or wide framing compared to the other two samples (confirmed: the tightest-framed shot in testing had the caption sit closer to the mouth than its own ideal position) -- that's the accepted tradeoff for a stable, non-jumpy result. If a video's framing varies dramatically shot-to-shot, sample more than 3 and/or bias the pick toward the more common framing rather than pure start/middle/end.

## Known limitations

- Glossary matching in `group_cards.py` is exact-phrase, not fuzzy -- it won't catch a mishearing that isn't already listed. Building the glossary is an ongoing, per-speaker task, not a one-time setup.
- **No reliable automated way to detect what *should* be in the glossary.** Fuzzy text-matching against domain vocabulary was tried and abandoned -- calibrated against real cases, it scored genuine mishearings *lower* than coincidental false positives (see step 2.5). This is a real, currently-unsolved gap, not a "todo, easy fix": catching a new mishearing category requires either the user noticing it, or Claude's own judgment while reading the transcript with `domain.json` in mind. A proper fix would need phonetic/sound-alike matching (e.g. a Polish-aware Soundex/Metaphone equivalent), which doesn't have a readily available library and wasn't built.
- Card grouping is a simple word-count/punctuation heuristic, not a prosody- or emphasis-aware one. Good enough as a draft; review before finalizing on anything that matters.
- Cut-aware grouping (`--timeline`) depends on `timeline.json` existing, which only rough-cut's `render.py` produces. Captioning a video edited some other way falls back to word-count/punctuation-only breaks, with no cut-awareness.
- No animation (pop-in, scale, karaoke-style word-by-word reveal) -- this burns in static per-card text. Would need per-word `\t` transform tags or per-word Dialogue lines timed individually; not implemented.
- **Font loading can fail silently** (see "Font resolution can fail silently" above) -- there is no automated check built into `burn_captions.py` itself that catches this; it requires manually running the fontselect verification whenever a font file changes. Forgetting this step means a styling session can proceed on a completely wrong font with no indication anything is off.
- **Vertical position is a fixed single value per video**, manually recomputed by sampling 3 frames (see "Vertical position" above) -- not adaptive within a video, not automated. A video with highly inconsistent framing between shots will have some shots positioned less precisely than others; this is an accepted, deliberate tradeoff (a per-shot tracking version was built and reverted for looking visually chaotic), not an oversight.

## File manifest

- `scripts/make_captions.py` -- word-level transcription of the edited video (`large-v3`).
- `scripts/flag_ambiguous.py` -- surfaces low-ASR-confidence words (the one mechanically reliable signal; see step 2.5 for why fuzzy domain-matching was removed).
- `scripts/group_cards.py` -- draft captions.json: glossary substitution + cut-aware card grouping.
- `scripts/burn_captions.py` -- captions.json -> `.ass` -> ffmpeg burn-in.
- `scripts/ffmpeg_util.py` -- shared ffmpeg binary resolution (own copy, kept self-contained from the rough-cut skill).
- `assets/TikTokSans-ExtraBold.ttf`, `assets/OFL.txt` -- font + license.
- `glossary.example.json` -- template to copy into a project as `glossary.json` (confirmed exact corrections, mechanical).
- `domain.example.json` -- template to copy into a project as `domain.json` (content niche + jargon, context for judgment, reusable per speaker).
